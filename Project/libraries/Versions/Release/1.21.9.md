### ${@name|Copper Chest}
* ${@name|Copper Chest}是一种新类型的有着涂蜡与氧化变种的箱子
* ${@name|Copper Chest}像铜块一样会随着时间氧化，并可以被涂蜡
* ${@name|Copper Chest}可以使用${@name|Chest}和铜锭合成

### ${@name|Copper Golem}
* ${@name|Copper Golem}是一种可以通过在铜块上放置南瓜灯或雕刻南瓜生成的生物。
* ${@name|Copper Golem}像铜块一样会随着时间氧化，并可以被涂蜡
* 如果${@name|Copper Golem}完全氧化并且没有被涂蜡，它将会变成${@name|Copper Golem Statue}方块
    * 当变成${@name|Copper Golem Statue}时它会掉落任何它手上拿着的物品
    * 变成${@name|Copper Golem Statue}的被命名牌命名的${@name|Copper Golem}会保留它的名字
* ${@name|Copper Golem}可以帮助将物品分类到箱子里
* 与${@name|Copper Golem}交互将会清除它手上的物品并使其掉落
* 铁傀儡有时会在最近的铜傀儡上种装饰花
  * 花可以使用剪刀剪下
  * 当铜傀儡变为铜傀儡像时花会掉落

#### ${@name|Copper Golem}物品分类
* 如果${@name|Copper Golem}没有持有物品，它将会寻找最近的${@name|Copper Chest}尝试拾取一个物品
    * ${@name|Copper Golem}会寻找任何氧化状态以及涂蜡状态变种来拾取物品
    * 它会同时拾取最多 16 个物品
    * ${@name|Copper Chest}中所有的物品都可以被拾取
* 如果${@name|Copper Golem}正拿着一个物品，它将会寻找最近的箱子尝试放进去
    * ${@name|Copper Golem}会寻找箱子以及陷阱箱尝试放入物品
    * 它会把物品放进空箱子里，或放进包含有着其手持的物品相同种类的物品的箱子里
* ${@name|Copper Golem}一次按顺序最多访问 10 个箱子或${@name|Copper Chest}来拾取或放下物品
* 如果没有找到匹配的箱子${@name|Copper Golem}将会闲置7秒后重试
* 箱子的搜索范围为以${@name|Copper Golem}的位置为中心，横向32格，垂直8格

### ${@name|Copper Golem Statue}方块
* ${@name|Copper Golem Statue}方块是一种有着涂蜡与氧化变种的新型装饰性方块
* ${@name|Copper Golem Statue}像铜块一样会随着时间氧化，并可以被涂蜡
* 可以对${@name|Copper Golem Statue}交互改变它的姿势
* 如果${@name|Copper Golem Statue}没有被氧化，对其使用斧头会将其变回${@name|Copper Golem}
* 每一种姿势都传输红石比较器信号

### ${@name|Shelf}
* ${@name|Shelf}是一种新型的装饰性方块，有着以下变种：
    * ${@name|Oak}
    * ${@name|Spruce}
    * ${@name|Birch}
    * ${@name|Jungle}
    * ${@name|Acacia}
    * ${@name|Dark Oak}
    * ${@name|Mangrove}
    * ${@name|Cherry|樱花}
    * ${@name|Pale Oak}
    * ${@name|Bamboo}
    * ${@name|Crimson|绯红}
    * ${@name|Warped|扭曲}
* ${@name|Shelf}可以通过六个相同去皮原木方块、去皮菌柄以及去皮竹块，填满顶行和底行合成
    * 这个配方生产 6 个${@name|Shelf}
* ${@name|Shelf}可储存最多三组物品：
    * 与其正面的任意3个槽位交互将会交换槽位中的物品与玩家手中的物品
    * ${@name|Shelf}在正面显示所有物品，与物品展示框相同
* 如果${@name|Shelf}被红石充能，其正面纹理会更改以反映其被充能；当它们被并排放置时，最多三个充能的${@name|Shelf}会被连接
    * 与单独一个充能的${@name|Shelf}交互会与玩家热键栏最右侧三个槽交换物品
    * 与两个充能的${@name|Shelf}交互会与玩家热键栏最右侧六个槽交换物品
    * 与三个充能的${@name|Shelf}交互会与玩家热键栏的所有物品
    * 其行为与玩家具体交互了哪个${@name|Shelf}无关

### Minecraft 服务端管理协议（MSMP）

增加了一个服务器管理 API (JSON-RPC over WebSocket) 用于专用服务器

* 此 API 默认禁用，可以由服务器 properties 文件启用

    * `management-server-enabled=true`
    * `management-server-host=localhost`
    * `management-server-port=25585`

* 当启用时 API 可以通过 `ws://<management-server-host>:<management-server-port>` 访问
* 支持查询以及更新服务器状态(玩家、白名单、操作员、设置、游戏规则)
* 在状态更改时发送通知 (如：玩家加入、游戏规则更新)
* 调用 `{"id":1,"method":"rpc.discover"}` 返回一个包含了当前运行的服务器支持的方法以及通知的 API schema
* 数据生成器在报表输出文件夹中提供一个 API schema (`json-rpc-api-schema.json`) 反映了 `rpc.discover` 方法返回的内容
* API 遵守 [JSON-RPC 2.0 规范](https://www.jsonrpc.org/specification)
* 使用带有命名空间的方法，并且保留的命名空间为 `minecraft` （如：`minecraft:players`,` minecraft:allowlist/add`）以及 `notification` （如 `notification:players/joined`）
    * 可以使用自定义命名空间以及额外方法进行扩展
* 核心方法组: 玩家、白名单、操作员、服务器(保存、停止)、服务器设置、游戏规则
* 方法调用示例:
    * 请求:

```json
{
  "method": "minecraft:allowlist/add",
  "id": 1,
  "params": [ [{ "name": "jeb_" }] ]
}
```

* 响应:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": [{ "id": "853c80ef-3c37-49fd-aa49-938b674adae6", "name": "jeb_" }]
}
```

* 通知示例:

```json
{
  "jsonrpc": "2.0",
  "method": "notification:players/joined",
  "params": [{ "id": "853c80ef-3c37-49fd-aa49-938b674adae6", "name": "jeb_" }]
}
```

* 错误示例:
    * 请求: `{"method": "minecraft:foo/bar","id": 1}`
    * 响应:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "jsonrpc": "2.0",
    "id": 1,
    "error": {
      "code": -32601,
      "message": "Method not found",
      "data": "Method not found: minecraft:foo/bar"
    }
  }
}
```

* 错误以及错误代码遵守 [JSON-RPC 2.0 错误对象格式](https://www.jsonrpc.org/specification#error_object)

### 聊天草稿

* 如果聊天被强制关闭，未发送的聊天信息现在会被存为聊天草稿
    * 这包含了被外边源关闭的情况，比如死亡或者打开了对话框，但是不包含玩家自己关闭了对话框

* 任何存在的聊天草稿将会在下一次聊天栏打开时提前填充到消息框
* 草稿文字有着灰色外观，直到其恢复
    * 可以通过交互使得恢复草稿文字为正常文字：
        * 键入更多文字
        * 使用方向键移动光标
        * 点击
        * 高亮
    * 在草稿未恢复时按下 Enter 键会像正常聊天消息一样发送出去
    * 在草稿未恢复时按下 Backspace 键会删除草稿


* 在聊天设置菜单中增加了一个选项，默认情况下保存未发送的聊天消息
    * 当该选项启用时，未发送的消息将会 _一直_ 保存为聊天草稿，即使玩家是自己关闭了聊天栏

### 服务器行为准则

当连接到服务器时，现在会显示服务器行为准则屏幕。

* 游玩服务器必须接受服务器行为准则
* 会提供一个检查框，勾选后不会再显示行为准则
    * 如果行为准则更改，它还会再次显示

#### 服务器设置

* 一个新的布尔字段 `enable-code-of-conduct` 添加入了 server.properties
    * 如果将其设为 `true` 将会使服务器在与 `server.properties` 所在文件夹中的名为 `codeofconduct` 子文件夹中寻找行为准则文件
    * 在文件夹中的每一个文件名应符合 `<language_code>.txt` 格式
        * 上方的语言将会尝试匹配玩家的语言
        * 如果不存在匹配玩家语言的文件，会使用 `en_us` 的行为准则
        * 如果 `en_us` 也不存在，则会匹配任意文件

## 更改

- 性能改进
- 锁链方块现在重命名为“铁链”
- 怪物刷怪蛋现在在难度设为和平时有一条提示说明
    - 当尝试在和平难度使用刷怪蛋时不再会短暂生成一刻
- 世界加载与“出生区区块”的更改
- 拥有任意玩家活动、被 forceload 的区块、激活的传送门或正在飞行的末影珍珠的维度现在会被视为“激活”，并会持续处理区块以及实体
    - 之前只有当玩家在该维度或某个区块被 forceload 时才会应用
- 重新设计了调试信息使其更少杂乱也更可配置
- 在当前服务器世界中的玩家现在一直会显示在社交屏幕中，即使他们离线
- 当从旁观者模式切换至创造模式时，如果玩家靠近地面，现在会停止飞行
    - 当玩家在其下方 1 格内有可碰撞表面（方块、快乐恶魂、船等）时被视为靠近地面
    - 当玩家在方块或可碰撞实体内部时飞行不会被禁用
        - 这用于防止玩家突然掉下
- 迁移了若干服务器属性到游戏规则
- 现在游玩游戏需要 OpenGL 3.3，从 3.2 提升。这不会改变我们之前发布的最低配置需求

### 性能改进

- 实体渲染顺序被改进以显著提高性能
    - 在有大量生物的区域（例如刷怪塔）你会注意到更流畅的游戏体验
    - 如果你的帧率瓶颈不是来自实体（例如高渲染距离且附近实体不多），FPS 不会有明显提升
- 在玩家视野外的粒子不再渲染，这在某些情况下会提供性能提升

### 世界加载与出生区区块

- 固定的“出生区区块”概念被移除
- 游戏会确保以下区块在玩家加入或服务器启动前被完整加载，其方式与之前出生区区块一致：
    - 被 forceload 命令加载的区块
    - 由于传送门活动而激活的区块
    - 被投掷的末影珍珠（仅单人）
    - 玩家附近的一小片区块（仅单人）
    - 创建世界时用于分配全局世界出生点的区块
- 世界加载界面现在显示玩家即将被放入的区块，而不是世界出生点周围的区块
- 加载界面现在有了一个进度条
    - 该进度条会计入上述所有类别的区块加载

### 服务器属性

为允许在服务器运行时更改设置，若干服务器属性被移除改为游戏规则。

- 移除服务器属性 `allow-nether`，改用新的游戏规则 `allowEnteringNetherUsingPortals`
- 移除服务器属性 `spawn-monsters`，改用新的游戏规则 `spawnMonsters`
- 移除服务器属性 `enable-command-block`，改用新的游戏规则 `commandBlocksEnabled`
- 移除服务器属性 `pvp`，改用新的游戏规则 `pvp`
- 增加服务器属性 `management-server-enabled` 控制是否启用 Minecraft 服务端管理协议
    - 默认 `false`
- 增加服务器属性 `management-server-port` 控制 Minecraft 服务端管理协议端口
    - 默认 `0`，每次启动分配可用端口
- 增加服务器属性 `management-server-host` 控制 Minecraft 服务端管理协议主机
    - 默认 `localhost`
- 增加服务器属性 `management-server-tls-enabled` 控制 Minecraft 服务端管理协议是否使用 TLS
    - 默认 `true`
- 增加服务器属性 `management-server-tls-keystore` 控制 TLS keystore 文件路径
    - 默认 空字符串
    - 当启用 TLS 且未提供 keystore 时专用服务器不会启动
- 增加服务器属性 `management-server-tls-keystore-password` 控制 TLS keystore 密码
    - 默认 空字符串
- 增加服务器属性 `status-heartbeat-interval` 控制管理服务器向已连接客户端发送心跳通知的间隔
    - 默认 0（禁用）
- 增加服务器属性 `enable-code-of-conduct`
    - 详情见“服务器设置”标题

### 游戏规则

- 移除 `spawnChunkRadius` 游戏规则
- 增加 `pvp` 游戏规则控制玩家是否可以对其他玩家造成伤害
    - 默认 `true`
- 增加 `allowEnteringNetherUsingPortals` 游戏规则控制下界是否可通过传送门进入
    - 默认 `true`
- 增加 `spawnMonsters` 游戏规则控制怪物是否自然生成
    - 默认 `true`
- 增加 `commandBlocksEnabled` 游戏规则控制命令方块是否启用
    - 默认 `true`
- 增加 `spawnerBlocksEnabled` 游戏规则控制刷怪笼是否启用
    - 默认 `true`

### 包格式

- 数据包与资源包的版本现在有次版本号
    - 游戏的包版本进行次版本号递增是向后兼容的，意味着该主版本之前迭代制作的所有包仍能工作
- 如果依赖次版本号中引入的资源，包可以对次版本号声明兼容性需求

#### 包元数据

pack.mcmeta 格式已更新：

- 移除了 `supported_formats` 字段
    - 如果你的包声明支持旧格式的包版本（数据包 < 82，资源包 < 65），仍然需要
    - 否则不允许，必须移除
- `pack_format` 字段现在可选
    - 如果你的包声明支持旧格式的包版本（数据包 < 82，资源包 < 65），仍然需要
- 新增必需字段 `min_format` —— 指定支持的最小版本
    - 完整版本由两个整数列表指定，例如 `[74, 1]`
    - 指定单个整数被解释为该主版本，例如 `74` 与 `[74, 0]` 相同
    - 指定仅一个整数的列表与直接指定该整数等价
- 新增必需字段 `max_format` —— 指定支持的最大版本
    - 完整版本由两个整数列表指定，例如 `[74, 1]`
    - 指定单个整数被解释为任意次版本，即次版本被视为 `0x7fffffff`
    - 指定仅一个整数的列表与直接指定该整数等价
- 对 overlay 条目：
    - 移除了 `formats` 字段
        - 如果你的包包含的任意 overlay 范围包含旧格式版本（数据包 < 82，资源包 < 65），则该字段对所有 overlay 定义仍然需要
        - 否则不允许，必须移除
    - 新增必需字段 `min_format` 与 `max_format`，其格式与上述字段相同

### 调试特性

- 一些之前在编译时被移除的调试特性现在被保留并可被玩家使用
- 可通过 JVM 属性启用单个值：
    - 首先必须使用 `-DMC_DEBUG_ENABLED` 或 `-DMC_DEBUG_ENABLED=true` 启用全局调试标记
    - 然后可启用指定特性，例如 `-DMC_DEBUG_BRAIN` 或 `-DMC_DEBUG_BRAIN=true`
    - 添加 `-DMC_DEBUG_PRINT_PROPERTIES` 可获取所有可用调试属性列表
- 警告：这是调试代码，主要意图是帮助模组作者——不用于正常游戏，务必谨慎！
    - 这些特性可能会崩溃你的游戏或破坏你的存档
    - 这些特性不保证按预期运行
    - 这些特性可能在未通知的情况下加入、修改或删除

### 玩家模型

增加了一种新的技术性实体，名为 `minecraft:mannequin`，仅能通过命令生成。

* 玩家模型是一个有着玩家外形但未与玩家连接的实体
* 玩家模型总是会在 `below_name` 记分栏位处显示一个“NPC”文本
* 玩家模型与活体功能相同——它们可以手持、装备物品，有属性与效果，会受到伤害等
* 玩家模型接受来自刷怪笼物品的 `minecraft:profile` 组件

数据字段：

* `profile` - 在实体上显示的档案。可以是玩家档案（与 `minecraft:profile` 组件格式相同），或一个有下列字段的玩家模型档案：
    * `texture` - 要使用的皮肤纹理的带有命名空间的 ID
        * 皮肤在以相对于 `texture` 文件夹的路径区分，带有 `.png` 后缀
        * 例如：`entity/player/wide/steve` 会使用默认史蒂夫皮肤
    * `cape` - 可选的要使用的披风的命名空间 ID
        * 指定时使用与 `texture` 字段相同格式
        * 缺省时玩家模型无披风
    * `elytra` - 可选的要使用的鞘翅的命名空间 ID
        * 指定时使用与 `texture` 字段相同格式
        * 缺省时使用披风对应的鞘翅纹理，如果披风也缺省则使用默认鞘翅纹理
    * `module` - 可选的模型类别，`wide` 或 `slim`
        * 缺省时使用 `wide`
* `hidden_layers` - 隐藏的皮肤层列表
    * 有效值：`cape`、 `jacket`、 `left_sleeve`、 `right_sleeve`、 `left_pants_leg`、 `right_pants_leg`、 `hat`
* `main_hand` - 玩家模型的主手——`left` 或 `right`

### 数据组件

#### 修改了 `minecraft:profile`

- 之前这个组件的内容会在档案被解析时永久的改变，例如在从 Minecraft 服务时
- 为了避免这个意料之外的更改的问题，现在解析值不再存储，并且档案内容会一直保持不被更改
- 这意味着这个组件现在有两个行为：
    - 静态: 当其存在 `properties` 字段时，或 `name` 和 `id` 均设置或均未设置时
        - 在这个状态下，档案会直接渲染
        - 如果 `properties` 字段缺省，会使用默认皮肤（由 `id` 决定，与离线模式相同）
        - 只意味着在组件创建时皮肤变为只读
        - 这个选项适合装饰用途

    - 动态: 当没有 `name` 或 `id` 时（但不是两个都没有）
        - 当包含这个档案的物品在客户端渲染时，档案将会被解析为最近期的值
        - 这意味着这个物品将会一直渲染为目标玩家的当前皮肤
        - 注：如果玩家在游戏运行时更改了皮肤，其更改不可见直到客户端重启

        - 在解析时可能会造成延迟，在这时物品会渲染为默认皮肤
        - 当存在时，物品名称将会替换档案名称
        - 只有 `id` 的物品会显示默认名称
        - 具有动态档案的物品会用提示栏与静态物品区分

- 物品堆叠基于精确的内容，所以即使动态组件与静态组件解析结果完全一样，它们也不会堆叠
- 在以前的地图上的未被新版本加载的玩家的头通常将会转为动态内容

### 着色器与后处理效果

> [!DEVNOTE] 虽然在资源包中可以做到，覆盖核心着色器被视为不受支持并非预期的资源包特性。这些着色器是游戏内部实现的一部分，因此可能会随着内部演进随时改变。我们理解覆盖核心着色器被用于很多很酷的资源包特性，其中很多缺乏受支持的替代方案。我们希望未来提供更好且受支持的替代方案。

#### 着色器

- 下列未使用的着色器被移除：
    - `core/position_color_lightmap.vsh`
    - `core/position_color_lightmap.fsh`
    - `core/position_color_tex_lightmap.vsh`
    - `core/position_color_tex_lightmap.fsh`
- 一个新的标记 `PER_FACE_LIGHTING` 被添加到下列着色器：
    - `core/entity.vsh`
    - `core/entity.fsh`
- 当设置时，正面和背面的基光照会被分别计算
- 由于现在需要 OpenGL 3.3，我们将所有着色器版本从 `150` bump 至 `330`

#### 全屏通道（Full Screen Passes）更改

- 下列顶点着色器被移除，用 `core/screenquad.vsh` 替换：
    - `core/blit_screen.vsh`
    - `post/blit.vsh`
    - `post/blur.vsh`
    - `post/invert.vsh`
    - `post/sobel.vsh`
    - `post/screenquad.vsh`
- 后处理效果以及光照贴图生成与全屏 blit 的顶点着色器不再传入 Position 属性，而是必须使用 `gl_VertexID` 指定顶点坐标
- 这些通道现在仅用 3 个顶点执行，而不是一个完整的四边形
